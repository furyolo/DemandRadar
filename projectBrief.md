# DemandRadar（需求雷达）— Project Brief

> 版本: v0.1 | 日期: 2026-06-18 | 状态: 规划中

---

## 1. 项目概述

DemandRadar 是一个自动化需求挖掘与商业评估系统。它以近期新闻热点为输入，借助 Smart Search 的 CLI + Skill 能力完成信息采集与深度分析，最终产出结构化的产品机会报告：包含用户画像、需求定义、市场规模估算和投入产出比评估。

一句话：**从新闻中持续发现值得做的产品机会。**

---

## 2. 核心原则：优先复用，拒绝重造

**每个模块在动手开发之前，必须先做一轮市场调研。** 这是本项目最基础的工程纪律，不是可选步骤。

### 调研流程

```
确定模块需求 → 搜索现有项目/Skill → 评估可复用性 → 决策
```

### 决策矩阵

| 情况 | 决策 |
|------|------|
| 有成熟开源项目且功能完全匹配 | **直接照搬**，不写一行新代码 |
| 有近似开源项目但需适配 | **Fork + 适配**，在既有代码基础上改 |
| 有商业产品 / Skill 可参考 | **参考其设计和交互**，用 Smart Search 实现 |
| 完全找不到类似项目 | **才从 0 开始做** |

### 具体执行规则

1. **每个模块（采集、清洗、分析、调研、输出）开发前**，先用 Smart Search + Exa 搜索一遍："有没有人已经做过类似的事？"
2. **群体调研部分**同理——如果已有成熟的 MCP Skill 或开源项目做市场调研，直接集成，不需要自己重写调研逻辑。
3. **发现竞品不是坏事**——竞品的存在验证了需求真实性；竞品的缺陷就是我们的切入点。
4. **禁止"Not Invented Here"心态**——拿过来能用的就是最好的代码。

## 3. 要解决的问题

产品经理和创业者在寻找下一个项目方向时，通常依赖个人经验、碎片化信息和直觉判断。这种方式有三个致命缺陷：

- **滞后性** — 等到看见别人做出来才跟进，窗口期已过
- **主观性** — 个人的"感觉很大"往往与真实市场规模差 1-2 个数量级
- **无系统性** — 今天看一篇 36Kr，明天刷一下 Twitter，没有持续的 pipeline

DemandRadar 要做的就是把这个过程系统化、自动化、量化。

## 4. 核心工作流

```
新闻采集 → 信息清洗 → 需求提取 → 群体评估 → 商业判断 → 输出 Brief
```

### 3.1 新闻采集

- 数据源：通过 Smart Search CLI 全网检索，限定时间窗口（默认近 30 天）
- 覆盖维度：科技、消费、政策、社会事件、出海、AI 应用
- 频率：每日定时抓取 + 可手动触发

### 3.2 信息清洗（Smart Search 内置过滤 + Skill 处理）

- 去重：同一事件多家媒体报道合并
- 去噪：过滤纯公关稿、软文、SEO 垃圾内容
- 分类标注：按行业 / 受众 / 热度等级自动打标

### 3.3 需求提取（Skill — 深度分析）

- 从每条热点中推导：**谁在焦虑？为什么焦虑？他们现在怎么解决的？**
- 输出格式：`[热点] → [人群] → [痛点] → [当前解决方案及其缺陷]`
- 合并同类需求，识别跨热点的共性痛点

### 3.4 群体规模评估（Smart Search 调研）

- 对每个候选需求，调研目标人群的规模：
  - 直接数据：行业报告、政府统计、平台 DAU
  - 间接数据：相关关键词搜索量、社区讨论量、竞品用户数
- 输出：TAM / SAM / SOM 三级估算 + 置信度标注

### 3.5 商业价值判断

- 付费意愿评估：当前解决方案的付费情况（客单价、付费率）
- 竞争格局：直接竞品、替代品、新进入者壁垒
- 投入产出比：粗略估算 MVP 开发成本 vs 预期收益
- 打分模型：需求强度 × 市场规模 × 付费意愿 × 可行性 → 综合分数

### 3.6 输出产品 Brief

- 每个高评分需求自动生成一份 Mini Product Brief：
  - 需求定义（一句话）
  - 目标用户画像
  - 核心功能列表（MVP 范围）
  - 市场规模引用及来源
  - 竞品简述
  - 推荐技术栈和关键风险
  - 投入产出比估算

## 6. 核心功能清单

### MVP（第一阶段）

- [ ] 新闻采集引擎：通过 Smart Search CLI 每日拉取 Top 100 热点
- [ ] 清洗管道：去重 → 分类 → 热度排序（Smart Search 内置能力 + Skill 补充）
- [ ] 需求提取 Agent：基于 Skill 的深度分析模版，从热点中推导用户需求
- [ ] 群体调研 Agent：自动检索目标人群规模数据
- [ ] 评分模型：多维度打分，产出 Top 10 需求排行榜
- [ ] 日报输出：每日自动生成「需求雷达日报」，包含 Top 3 高价值需求 + Mini Brief
- [ ] CLI 交互：终端内浏览、筛选、深入查看某条需求的完整分析

### 第二阶段

- [ ] Web Dashboard：可视化趋势变化、需求热度曲线
- [ ] 历史对比：追踪某条需求从"信号"到"热点"的演进过程
- [ ] 个性化配置：自定义关注领域、排除行业、调整评分权重
- [ ] 邮件 / Slack / 飞书推送

### 第三阶段

- [ ] 需求预测：基于历史数据预测下一周可能爆发的需求方向
- [ ] 竞品深度分析：自动抓取竞品融资、产品迭代、用户评价
- [ ] 协作功能：团队内对需求进行标注、讨论、投票

## 7. 技术架构概要

```
┌─────────────────────────────────────────────┐
│                  CLI / Web UI                │
├─────────────────────────────────────────────┤
│              调度层（Cron / Trigger）         │
├──────────┬──────────┬───────────┬───────────┤
│ 采集模块  │ 清洗模块  │ 分析模块   │ 输出模块   │
│          │          │           │           │
│ Smart    │ Skill    │ Skill     │ Brief     │
│ Search   │ Pipeline │ Agents    │ Generator │
├──────────┴──────────┴───────────┴───────────┤
│              数据层（SQLite / JSON）          │
└─────────────────────────────────────────────┘
```

- **语言**：TypeScript / Python（与现有 Smart Search 工具链保持一致）
- **存储**：SQLite 存结构化数据 + Markdown 文件存 Brief 文档
- **调度**：Cron job 每日触发主流程
- **输出格式**：Markdown（兼容任意编辑器）+ JSON（可编程消费）

## 8. 成功指标

| 指标 | 目标值（3 个月后） | 衡量方式 |
|------|------------------|---------|
| 日均可产出需求数 | ≥ 30 条 | 系统统计 |
| 高评分需求占比（>70 分） | ≥ 20% | 评分模型输出 |
| 需求到产品转化率 | ≥ 1 个实际启动 | 人工跟踪 |
| 误报率（需求被判定为伪需求） | ≤ 15% | 周度人工抽检 |
| 单次完整运行耗时 | ≤ 30 分钟 | 系统计时 |

## 9. 风险与挑战

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 新闻源质量不稳定 | 输入噪音大，产出质量下降 | 多源交叉验证 + 人工抽查反馈循环 |
| 市场规模数据难以获取 | 评分模型失准 | 标注置信度；对无法验证的数据降低权重 |
| LLM 分析幻觉 | 产出虚构的需求或数据 | 所有数据引用强制附加来源 URL；人工抽检 |
| 热点生命周期短 | 分析完已过气 | 缩短 pipeline 耗时；优先分析长期趋势型热点 |

## 附录 A：市场调研记录（2026-06-18）

> 调研目的：确认 DemandRadar 各子模块是否有现有项目/Skill 可直接复用，避免从零开发。

### A.1 可直接复用或高度参考的项目

#### 核心逻辑（需求挖掘 + 评分）

**Pain-Point-Miner** — GitHub 开源，最接近 DemandRadar 核心逻辑。
- 从 Reddit 抓帖 → AI 提取痛点（痛苦强度、紧迫性、货币化评分、市场成熟度、情绪、预算信号、已尝试方案）
- Embedding + PGVector 语义聚类，自动发现需求簇
- 多因子加权打分：Pain Intensity (%) + Urgency (%) + Monetization (30%)
- 蓝海/红海市场成熟度修正分
- **复用建议**：核心 pipeline（提取→嵌入→聚类→评分）可直接 Fork 改造，替换 Reddit 数据源为 Smart Search 新闻源

**GoldVein.ai** — 已做成 Claude 可加载模块。
- 8 源并行扫描（Reddit, HN, IH, GitHub, Reviews, Google, YouTube, PH）
- 交叉信号评分：单源信号不理，4 源以上收敛才输出
- 输出 7 天行动计划
- **复用建议**：形式最接近我们的目标——证明"Claude Code 内运行需求挖掘"完全可行。可参考其多源交叉验证评分逻辑

#### 采集与清洗

**SCOOP** — GitHub 开源，CLI 驱动新闻采集平台。
- CLI 采集（`scoop ingest`）+ 规范化 + 嵌入 + 语义去重
- Go 后端 + React 前端 + Python 嵌入服务
- 自动将相似文章归并为 Story
- **复用建议**：去重和入库逻辑可直接参考；CLI 设计模式可照搬

**Market Intelligence Radar** — GitHub 开源，Python 全栈。
- 新闻源配置化扫描 + Firecrawl 抓取 + LLM 提取结构化情报
- Next.js Dashboard + PostgreSQL
- **复用建议**：采集管道架构清晰，scanner → scraper → extractor → storage 模式可直接参考

#### 信号分类

**Signal Hunter** — GitHub 开源，OpenClaw 插件。
- 10 种信号类型分类体系：pain_point, feature_request, bug_report, adoption_signal, comparison, use_case, pricing_concern, positive_feedback, market_observation, security_concern
- Qdrant 语义搜索 + bge-m3 嵌入
- 日频自动采集
- **复用建议**：信号分类体系可直接照搬，作为需求提取 Agent 的输出 schema

#### 输出与简报

**Scout OSS** — GitHub 开源，本地研究 Agent。
- Mission-driven 研究 + Markdown Brief 输出
- CLI + Skill 双模式，去重记忆
- **复用建议**：输出格式和 Agent 调度模式可直接参考

**TrendingHunter** — GitHub 开源，3 阶段 LLM Pipeline。
- Draft → Audit → Rewrite 三阶段 LLM pipeline
- 生成结构化 Markdown 报告
- **复用建议**：多阶段 LLM 验证模式可提高输出质量，避免单次 LLM 幻觉

### A.2 商业竞品（验证市场需求）

| 产品 | 价格 | 差异化空间 |
|------|------|-----------|
| Trendstr.io | 付费 | 仅覆盖英文社区，无中文源 |
| Trendi (gettrendi.io) | 免费+付费 | 侧重内容创作者，非产品经理 |
| TrendOS | $29/月起 | 偏向趋势预测，非需求→产品转化 |
| Superway.ai | 企业付费 | 企业级，太重；个人开发者用不上 |
| Trending Shield | 付费 | 偏向营销/电商，非通用需求挖掘 |

**关键发现**：英文互联网社区（Reddit/HN/PH/GitHub/Indie Hackers）仍是 MVP 阶段最高信号密度的数据源，竞品和开源项目也主要围绕这些源沉淀。中文互联网（知乎/即刻/小红书/微博/36Kr）保留为后续差异化扩展方向。
